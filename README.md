# Система распределения обращений клиентов между сотрудниками
Cистема, которая выполняет распределение обращений клиентов между сотрудниками технической поддержки для ускорения времени обработки запросов, а следовательно и повышения уровня сервиса.

## Фунционал
* Присутствует возможность создания, просмотра, редактирования и удаления клиентов, обращения которых будут распределяться. 
Клиенты могут быть обычными либо VIP. VIP-статус клиента повышает 
приоритет обращения этого клиента во время распределения;
* Возможность создания, просмотра, редактирования и удаления сотрудников, которые будут назначаться на обращения для обработки;
* Каждый сотрудник может либо находиться в статусе «работа», когда на него могут назначаться обращения, либо в статусе «перерыв», когда система не 
будет назначать на него обращения;
*  У каждого сотрудника определяется максимальный лимит одновременно назначенных активных  обращений, достигнув которого сотрудник перестает участвовать в процессе 
распределения;
* Присутствует возможность создания, просмотра, редактирования и удаления команд сотрудников. Сначала определяется команду, ответственная за обращение, а потом на обращение назначается один 
из сотрудников этой команды. Каждый сотрудник может принадлежать нескольким командам;
* Присутствует функционал создания, просмотра, редактирования и удаления тем и подтем, с которыми будут связываются команды сотрудников;
* Поддерживается возможность создания обращения от лица клиента с указанием темы, подтемы и текста обращения. После создания обращения оно распределяется на одного из сотрудников;
* Возможно закрыть активное обращение. Закрытое обращение не участвует в алгоритме распределения, а также не считается обрабатываемым сотрудником;
* Реализован функционал автоматизаций, которые в зависимости от условий будут устанавливать команду сотрудников обращения. Автоматизации поддерживают следующие условия:
   - равенство почты клиента указанному значению;
   - время создания обращения в указанный временной промежуток;
   - содержание указанных ключевых слов в тексте обращения;
   - равенство темы обращения указанному значению.
* Все условия в рамках автоматизации могут быть объединены логическими «и» или «или». Автоматизация завершается после прохождения всех условий либо при ложном значении одного из условий;
* Система позволяет просматривать созданные обращения пользователей, динамически обновлять состояния обращения для корректного отображения результатов распределения;
* Система отображает, сколько еще обращений может быть назначено на определенного сотрудника. 

## Особенности
* Серверная часть приложения поддерживает асинхронный формат работы для повышения производительности - после создания обращения оно асинхронно распределяется; 
* Приложение развертывается с помощью Docker Compose.

## Алгоритм распределения
Распределение состоит из трех этапов:
1. Исходя из темы,  подтемы, а также VIP-статуса клиента, создавшего обращение, осуществляется поиск команды сотрудников, которая будет обрабатывать обращение;
<div align="center"><img width="481" height="674" alt="image" src="https://github.com/user-attachments/assets/acf70e9a-b17b-4cf4-a22e-d5b40f65612d" /></div>
2. Далее для обращения выполняются настроенные пользователями автоматизации. Автоматизация состоит из последовательных шагов. Каждый шаг может 
представлять собой либо условие, либо действие по изменению команды обращения; <br>
3. На третьем этапе из выбранной команды выбирается сотрудник, который 
и будет обрабатывать обращения. Данный этап является последним в цепочке распределения, для ускорения 
работы системы он должен работать в фоновом режиме (запускаться по RabbitMQ ивенту), запускаться при 
получении асинхронного сообщения.
Финальный этап должен решать 3 задачи:
<br>- необходимо обработать асинхронное сообщение, рассчитать 
приоритет обращения и переместить его в очередь на распределение;
<br>- нужно получать обращения из очереди на распределение, для 
каждого из них требуется находить оптимального сотрудника;
<br>- необходимо обновлять состояние обращения, назначая на него 
найденного сотрудника.
<br>Для повышения эффективности все 3 этих этапа работают через асинхронные задачи, реализованные через asynq. У каждого сотрудника есть ячейки, кол-во которых равно лимиту сотрудника. При назначении на сотрудника обращения
занимает одну из ячеек. Сотрудник перестает участвовать в процессе распределения, когда все его ячейки заняты.

## Симуляция данных
Сам механизм распределения полностью реализуется бэкендом. Однако для ее демонстрации, а также тестирования был сделан фронтенд, а также на стороне бэкенда был подготовлен функционал управления сущностями, которые участвуют в процессе распределения. 
Фронтенд использует поллинг - автоматически обновляет страницы со статусом распределения раз в 3 секунды для демонстрации актуального прогресса.

## Используемые технологии
* **Backend:** Go, asynq;
* **Frontend:** TypeScript, React;
* **Хранение данных:** PostrgreSQL, React (для хранения асинхронных задач);
* **Брокер сообщений:** RabbitMQ;
* **Развертывание:** Docker compose.

## Инструкция по запуску
1. Выполнить: `docker compose up`;
2. Перейти на `localhost:3000`.
