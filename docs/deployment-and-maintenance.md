# Раздел: Развертывание и сопровождение

## Общая схема контейнерного развертывания

В рамках дипломного проекта выбрана модель развертывания на базе Docker Compose. Такой подход позволяет запускать все части системы как единый стек, сохранить воспроизводимость окружения и сократить время ввода решения в эксплуатацию.

Точка оркестрации: `compose.yaml` в корне репозитория.  
Стек поднимается командой:

```bash
docker compose up --build
```

В конфигурации используются следующие сервисы:

- `frontend` - пользовательский web-интерфейс (React/Vite, раздача статических файлов).
- `crud` - backend-сервис с REST API (Go), выполняющий бизнес-логику.
- `postgres-crud` - реляционная база данных PostgreSQL.
- `redis` - in-memory хранилище и инфраструктурный компонент для фоновых задач.
- `rabbitmq` - брокер сообщений для событийного взаимодействия.

Все контейнеры подключены к общей пользовательской сети `app-network` (`bridge`), что обеспечивает корректную сервисную маршрутизацию по именам контейнеров (`postgres-crud`, `redis`, `rabbitmq`).

## Роль фронтенда в контейнерной схеме

Сервис `frontend` собирается из каталога `frontend` и публикуется на порту `3000`.  
Фронтенд зависит от `crud` через:

- `depends_on` с условием `condition: service_healthy`.

Это означает, что запуск фронтенда происходит после того, как backend пройдет проверку здоровья.  
В браузере frontend обращается к API по адресу, задаваемому через `VITE_API_URL` при сборке образа.

С практической точки зрения контейнер фронтенда выполняет две задачи:

1. На этапе сборки формирует production-бандл (`dist`).
2. На этапе выполнения раздает статические файлы пользователю.

## Роль бэкенда в контейнерной схеме

Сервис `crud` собирается из каталога `CRUD-service`, работает на порту `8080` и получает параметры через переменные окружения, заданные в `compose.yaml`:

- `DATABASE_URL` - подключение к PostgreSQL.
- `POSTGRES_DSN` - DSN для подсистемы балансировки (в текущем проекте совпадает с `DATABASE_URL`).
- `SERVER_ADDR` - адрес HTTP-сервера.
- `ROLE` - роль запускаемого процесса.
- `REDIS_ADDR` - адрес Redis.
- `RABBIT_URL` и `RABBIT_QUEUE` - параметры подключения к RabbitMQ.
- `BATCH_SIZE` и `BATCH_TIMEOUT_MS` - параметры пакетной обработки событий.
- `MATCHER_TICK_MS` - период фонового матчера.
- `FETCH_APPEALS_LIMIT` и `FETCH_MANAGERS_LIMIT` - лимиты выборки для фоновых операций.

Использование `environment` в Docker Compose является практичным решением для сопровождения, потому что:

1. **Конфигурация централизована** - параметры запуска видны в одном месте вместе с описанием сервиса.
2. **Поведение среды воспроизводимо** - одинаковые значения легко использовать на разных машинах (разработчик, тест, демо).
3. **Упрощается изменение настроек без пересборки кода** - достаточно изменить переменные в Compose.
4. **Дефолты фиксируются явно** - даже если в коде есть значения по умолчанию, в Compose видно, какие значения проект использует фактически.

При старте контейнера backend выполняет миграции (`goose ... up`) и только после этого запускает основное приложение (`./main`).  
Такой порядок исключает ситуацию, когда сервис стартует на неподготовленной схеме БД.

Также `crud` объявляет зависимости от:

- `postgres-crud` (service_healthy),
- `redis` (service_healthy),
- `rabbitmq` (service_healthy).

Это повышает надежность старта и уменьшает количество ошибок инициализации.

## Внешние зависимости и их назначение

### PostgreSQL (`postgres-crud`)

Основное хранилище предметных данных (обращения, сотрудники, слоты и др.).  
Порт на хосте: `5435 -> 5432`.

### Redis (`redis`)

Быстрый инфраструктурный компонент для задач, где требуется оперативный доступ к данным в памяти.  
Порт на хосте: `6379 -> 6379`.

### RabbitMQ (`rabbitmq`)

Брокер сообщений для асинхронных сценариев и интеграции событий.  
Порты на хосте:

- `5674 -> 5672` (AMQP),
- `15674 -> 15672` (web-интерфейс управления).

## Volumes и их роль

В проекте используются именованные тома Docker:

- `postgres_crud_data:/var/lib/postgresql/data`
- `redis_data:/data`
- `rabbitmq_data:/var/lib/rabbitmq`

Назначение томов:

1. **Сохранение состояния между перезапусками контейнеров.**  
   Без томов данные БД, очередей и кэша были бы потеряны при удалении контейнера.
2. **Изоляция данных от жизненного цикла образа.**  
   Образ можно пересобрать, но данные в томах сохраняются.
3. **Упрощение сопровождения и восстановления.**  
   Можно делать резервирование и перенос данных на уровне томов.

Таким образом, volumes обеспечивают практическую устойчивость системы в условиях эксплуатации.

## Healthchecks и контроль готовности сервисов

Healthcheck - это встроенный механизм Docker для проверки работоспособности контейнера по заданному сценарию.

В рассматриваемом проекте:

- `crud`: HTTP-проверка `GET /health` через `wget`.
- `postgres-crud`: `pg_isready`.
- `redis`: `redis-cli ping`.
- `rabbitmq`: `rabbitmq-diagnostics check_port_connectivity`.

Параметры `interval`, `timeout`, `retries`, `start_period` задают периодичность, таймаут, число попыток и стартовое окно инициализации.

Практический эффект healthchecks:

1. Compose корректно определяет состояние сервиса (`healthy`/`unhealthy`).
2. Зависимые контейнеры стартуют только после готовности нужных компонентов.
3. Упрощается диагностика при сопровождении (по статусу видно, какой сервис не готов).

## Многоэтапная сборка: как работает и зачем нужна

### Общий принцип multi-stage build

Многоэтапная сборка (multi-stage build) позволяет разделить процесс создания образа на несколько этапов:

1. **Builder stage** - устанавливаются зависимости и выполняется сборка.
2. **Runtime stage** - в финальный образ копируется только необходимый результат сборки.

Преимущества подхода:

- уменьшение размера итогового образа,
- сокращение поверхности атаки (меньше инструментов в runtime),
- ускорение доставки и запуска в production.

### Реализация в проекте (frontend)

Во `frontend/Dockerfile` используется двухэтапная схема:

- `FROM node:20-alpine AS builder` - установка npm-зависимостей и `npm run build`;
- `FROM node:20-alpine` - легковесный runtime-контейнер, куда копируется только `/app/dist`, далее статика отдается через `serve`.

Это корректная реализация multi-stage для frontend-части.

### Примечание по backend

В `CRUD-service/Dockerfile` сейчас используется одноступенчатая сборка: компиляция и запуск происходят в одном образе `golang`.  
Для дальнейшего развития проекта целесообразно перейти на multi-stage и выносить скомпилированный бинарный файл в отдельный минимальный runtime-образ. Это позволит дополнительно уменьшить размер backend-образа и улучшить эксплуатационные характеристики.

## Сопровождение в процессе эксплуатации

Для раздела сопровождения в рамках дипломного проекта важны следующие практики:

1. **Стандартизированный запуск**
   - `docker compose up --build` для первичного старта;
   - `docker compose up -d` для фонового запуска.
2. **Мониторинг состояния**
   - `docker compose ps` для контроля статусов и health;
   - `docker compose logs -f <service>` для анализа ошибок.
3. **Управление жизненным циклом**
   - `docker compose restart <service>` для точечного перезапуска;
   - `docker compose down` для штатной остановки.
4. **Работа с данными**
   - при `docker compose down -v` удаляются контейнеры вместе с томами, что эквивалентно полной очистке состояния;
   - данную операцию следует применять только в тестовых сценариях или при осознанной необходимости сброса данных.

## Вывод по разделу

Контейнерная схема развертывания данного проекта обеспечивает воспроизводимость, модульность и предсказуемость эксплуатации.  
Использование Docker Compose, healthchecks, зависимостей по состоянию сервисов и постоянных volumes формирует основу для стабильной работы системы.  
Применение многоэтапной сборки во frontend подтверждает практическую направленность решения и соответствует требованиям современного DevOps-подхода к сопровождению программных систем.
