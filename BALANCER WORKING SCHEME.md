Группируем ивенты по APPEAL_CLOSED или APPEAL_NEEDS_DISTRIBUTION
Для каждой группы помещаем в отдельную очередь (в зав-ти от имени группы) джобу для обработки (NewBatchUpdateTask).

======Отдельный воркер под batchUpdate джобы======
Проверяем, что обращение не закрыто, если закрыто, убираем его из очереди на распределение.
Рассчитываем обновленный приоритет обращения (base * 1000 + ageMinutes * 10) base = 2 если ВИП-клиент, иначе base = 1
Добавляем значение в очередь для распределения или обновляем существующее значение

======В matсher.go воркер который каждую секунду запускается======
Получает текущие обращения из очереди (ORDER BY p.priority DESC, p.updated_at ASC)
Получает текущих сотрудников со статусом working, которые при этом имеют меньше всего активных обращений. При равном количестве обращений дольше не получали назначение (В рамках одного ORDER BY).
Для каждого сотрудника получает список свободных слотов, отсортированных по времени последнего назначения по возрастанию

Для всех сотрудников получаем последнее время назначения, кол-во активных обращений
Формируем map, где для каждой тимы лежат сотрудники определенные
Сортируем список сотрудников для каждой команды:
1. Если у сотрудников разное кол-во обращений, то в начало ставится сотр с меньшим кол-вом
2. Иначе в начало ставится на которого раньше назначили последнее обращение
3. Иначе в начало ставится сотрудник у которого доступно больше слотов
Для каждого обращения из очереди выбираем сотрудника c наименьшим количеством активных обращений, на которого дольше всего не назналось обращений.
Для оптимального сотрудника выбираем слот, который дольше всего не был занят


======Воркер, занимающийся назначением======
Назначает сотрудника на обращение в отдельном потоке.
Защита от гонок заключается в блокировке на уровне базы данных
